# MEMORY SYSTEM — GEN-2 ARCHITEKTUR (Verbindliche Anleitung)

Dies ist das vollständige Gedächtnissystem für Wanda und alle Agenten. Es ist kein optionales Feature — es ist die Infrastruktur, auf der alle Entscheidungen und Aktionen basieren. Jeder Agent muss es kennen und aktiv nutzen.

## Warum: Das Problem mit vollem Kontext (Context Rot)

Modelle verlieren bei überfülltem Kontext ihre Genauigkeit von 98% auf 64%. Vier Ursachen:
- **Context Poisoning** — Halluzination wird als Fakt behandelt und verstärkt sich
- **Context Distraction** — Überflüssige Details überdecken die Kerninstruktion
- **Context Confusion** — Widersprüchliche Infos führen zu inkonsistenten Aktionen
- **Context Clash** — Widersprüche erzeugen logische Sackgassen

Die Lösung: externes, strukturiertes Gedächtnis statt volles Kontextfenster. Statt 20.000 Token Historien zu laden, lädt der Agent 500 Token hochrelevante Fakten aus PROJECTS.md und TACIT_KNOWLEDGE.md.

## Schicht 1: PARA-Struktur (Dateisystem-Fundament)

```
PARA/
├── Projects/
│   └── PROJECTS.md          ← Boot-Kontext (P1–P4, max 500 Token, täglich gültig)
├── Areas/
│   ├── TACIT_KNOWLEDGE.md   ← Bi-temporale Fakten (Präferenzen, Lektionen, Muster)
│   ├── STANDARDS.md         ← Maschinenlesbare Systemregeln (Coding, Git, Deploy)
│   └── INFRASTRUCTURE.md    ← Technische Infrastruktur-Fakten
├── Resources/
│   └── [thema].md           ← Detailliertes Research, API-Docs, Referenzen
├── Archives/
│   └── daily/               ← Archivierte L1-Logs (kalt, nur auf Anfrage)
└── memory/
    └── daily/
        └── YYYY-MM-DD.md    ← Episodische Tages-Logs (aktiv, L1)
```

**Pflicht:** Kein Wissen existiert außerhalb dieser Struktur. Kein Agent speichert Wissen nur "im Chat".

## Schicht 2: Bi-temporales Fakten-Format

Jeder Fakt in TACIT_KNOWLEDGE.md und im Knowledge Graph trägt zwei Zeitstempel:
- **Valid Time:** Wann galt das Ereignis in der realen Welt?
- **Transaction Time:** Wann wurde es erfasst?

**Format:** `- [Fakt als Satz] | Valid: YYYY-MM-DD | Status: active`

**Bei Widerspruch:** `- [alter Fakt] | Valid: YYYY-MM-DD | Status: deprecated | Superseded: YYYY-MM-DD`

Niemals löschen. Immer versionieren. Die Fakten-Evolution muss vollständig rekonstruierbar bleiben.

## Schicht 3: Episodisches Daily Logging (Observation Masking)

**Jede Session schreibt in:** `PARA/memory/daily/YYYY-MM-DD.md`

**Format (Observation Masking):**
```
## HH:MM — [Kurz-Label]
[Action] → [Observation (Essenz, kein roher Tool-Output)] → [Insight]
```

**Beispiel:**
```
## 09:14 — QMD Setup
[Action] → qmd install + 4 Collections konfiguriert → [Insight] QMD ist production-ready: BM25 + Vector in einem Tool ohne separaten Vector-Server
```

Rohe Tool-Outputs (80% des Kontextes) werden niemals gespeichert — nur die Essenz. Das spart ~50% Tokens ohne Informationsverlust.

Der `session-stop.sh`-Hook schreibt automatisch bei Session-Ende:
```
[Session-Stop] → [Tool-Calls: N | Daily-Note: N Zeilen] → [Context gesichert]
```

## Schicht 4: Knowledge Graph MCP (Bi-temporal, SQLite)

**Auto-verfügbar via:** `settings.json → mcpServers.knowledge-graph`

Der Knowledge Graph ist der semantische Index aller Fakten. Er synchronisiert sich mit TACIT_KNOWLEDGE.md und ist in jeder Session ohne Dateizugriff abfragbar.

**5 MCP-Tools:**

| Tool | Wann nutzen |
|------|-------------|
| `kg_add_fact(fact, category, valid_from)` | Neue Erkenntnis, Entscheidung, gelernter Fakt |
| `kg_search(query, category)` | Präferenzen, Systeminfos, Muster abrufen |
| `kg_temporal_query(subject, as_of_date)` | Was war an Datum X über Thema Y bekannt? |
| `kg_deprecate(fact_id, reason)` | Veralteten Fakt als deprecated markieren |
| `kg_ingest_para()` | TACIT_KNOWLEDGE.md bulk-importieren |

**Kategorien:** `preferences | patterns | infrastructure | lessons | research`

**Upgrade-Pfad (Phase 2):** graphiti-core + Neo4j/FalkorDB → Multi-Hop-Reasoning, inkrementelle Echtzeit-Updates, 90% weniger Latenz. Voraussetzung: `ANTHROPIC_API_KEY` in `settings.json → env`.

## Schicht 5: Semantische Suchschicht (QMD + Qdrant)

**QMD** (`qmd mcp` via `settings.json → mcpServers.qmd`):
- 5 Collections: `projects | areas | resources | archives | episodic`
- Hybrid-Suche: BM25 (Keyword) + Vector (Semantisch) + Exakt
- Indiziert alle PARA-Markdown-Dateien
- Nightly: `qmd update && qmd embed` (re-scannt Filesystem, aktualisiert Vektoren)

**Qdrant** (Docker, `localhost:6333`):
- Collection `para_knowledge`, 768 Dims, Cosine Similarity
- Embeddings: `all-mpnet-base-v2` via `embed_para.py --incremental`
- Nur für semantische Suche (Stufe 3 der Kaskade)
- Start: `cd PARA && docker compose up -d`

## Schicht 6: Retrieval-Kaskade (Verbindlich)

Vor jeder Antwort / vor jedem Code-Schreiben:

```
1. ripgrep        → exakte Suche im lokalen Workspace (Code, Logs, Configs)
2. QMD BM25       → Keyword-Suche in PARA Markdown-Docs
3. Qdrant Vector  → semantische Suche bei vagen Anfragen
4. Knowledge Graph → zeitliche Beziehungen, Fakten-Evolution
```

Das vollständige Laden von Dateien ohne Kaskade ist verboten.

## Schicht 8: Nightly Consolidation — Summarization Tree (02:00 Cron)

```
Cron: 0 2 * * *   →   /home/jannis/scripts/nightly-consolidation.sh
```

**Ablauf:**
1. **L1 archivieren** — `memory/daily/GESTERN.md` → `Archives/daily/`
2. **L1→L2 verdichten** — `[Insight]`-Blöcke extrahieren → bi-temporal in `TACIT_KNOWLEDGE.md → Learned Lessons`
3. **Neuen L1-Log anlegen** — `memory/daily/HEUTE.md` mit leerem Observation-Masking-Template
4. **QMD re-indexieren** — `qmd update && qmd embed` (alle 5 Collections)
5. **Qdrant inkrementell** — nur geänderte Dateien neu einbetten (`--incremental`)
6. **KG synchronisieren** — neue TACIT-Fakten in SQLite-DB importieren (Duplikat-Check)

**Vollständige Summarization-Hierarchie (TiMem-Modell):**

| Level | Inhalt | Datei | Frequenz |
|-------|--------|-------|----------|
| L1 | Rohe Tages-Logs | `memory/daily/YYYY-MM-DD.md` | Kontinuierlich |
| L2 | Verdichtete Insights | `TACIT_KNOWLEDGE.md` | Nightly |
| L3 | Themen-Cluster | `Resources/[thema].md` | Wöchentlich |
| L4 | Verhaltens-Muster | `STANDARDS.md` | Wöchentlich |
| L5 | Persona / Strategie | `PROJECTS.md` P1–P4 | Monatlich |
